= API Differences between @asciidoctor/core and @asciidoctor/native
:toc: left
:toclevels: 3
:icons: font

This document lists the differences between the `@asciidoctor/core` package (Opal-based, synchronous) and the `@asciidoctor/native` package (native JavaScript, asynchronous).

== Major architectural changes

=== Execution model: synchronous → asynchronous

The most fundamental difference is the shift from a synchronous to an asynchronous API.
All operations involving I/O (file reads, network requests, full parsing) return a `Promise`.

[cols="1,1,2"]
|===
| core (`@asciidoctor/core`) | native (`@asciidoctor/native`) | Note

| `asciidoctor.load(source, opts)` → `Document`
| `asciidoctor.load(source, opts)` → `Promise<Document>`
| Must be awaited

| `asciidoctor.loadFile(path, opts)` → `Document`
| `asciidoctor.loadFile(path, opts)` → `Promise<Document>`
| Must be awaited

| `asciidoctor.convert(source, opts)` → `string`
| `asciidoctor.convert(source, opts)` → `Promise<string>`
| Must be awaited

| `asciidoctor.convertFile(path, opts)` → `string`
| `asciidoctor.convertFile(path, opts)` → `Promise<string>`
| Must be awaited

| `document.parse(data?)` → `Document`
| `document.parse(data?)` → `Promise<Document>`
| Must be awaited

| `document.convert(opts?)` → `string`
| `document.convert(opts?)` → `Promise<string>`
| Must be awaited
|===

=== Removal of the Opal dependency

`@asciidoctor/core` relies on Opal (a Ruby-to-JavaScript transpiler).
`@asciidoctor/native` is pure JavaScript (ESM) with no transpilation dependency.

=== No instantiation required

`@asciidoctor/core` exports a factory function that must be called to obtain an Asciidoctor instance.
`@asciidoctor/native` exports a ready-to-use object — no factory call is needed.

[cols="1,1"]
|===
| core | native

| `require('@asciidoctor/core')()` → must call the factory to get an instance
| `import asciidoctor from '@asciidoctor/native'` → object is immediately usable

| `asciidoctor.convert(...)` after instantiation
| `asciidoctor.convert(...)` directly after import
|===

=== Named exports

In addition to the default export, `@asciidoctor/native` exposes every public member as a named export so that consumers can import only what they need.

.Functions
[source,javascript]
----
import { load, convert, loadFile, convertFile, getVersion, getCoreVersion } from '@asciidoctor/native'
----

.Classes and objects
[source,javascript]
----
import {
  Extensions,
  ConverterFactory,
  Html5Converter,
  SyntaxHighlighter,
  SyntaxHighlighterBase,
  Block,
  Section,
  LoggerManager,
  MemoryLogger,
  NullLogger,
  SafeMode,
  Timings,
} from '@asciidoctor/native'
----

NOTE: `loadFile` and `convertFile` are not available as named exports in the browser build.

=== Strict camelCase naming

In `@asciidoctor/core`, some internal properties and options retained the Ruby `snake_case` style.
`@asciidoctor/native` uses strict camelCase for all instance properties.

[cols="1,1"]
|===
| core (inherited `snake_case`) | native (`camelCase`)

| `node_name`
| `nodeName`

| `source_location`
| `sourceLocation`

| `content_model`
| `contentModel`

| `default_subs`
| `defaultSubs`
|===

== Changes by class

=== Asciidoctor (top-level object)

[cols="1,1,2"]
|===
| core | native | Note

| `getVersion()` → `string`
| `getVersion()` → `string`
| Returns the `@asciidoctor/native` package version

| `getCoreVersion()` → `string`
| `getCoreVersion()` → `string`
| Returns the Asciidoctor Ruby core version

| `getRuntime()` → `Object`
| _(not implemented)_
| Returns platform / engine / GEM_HOME info — no equivalent in native
|===

=== AbstractNode

==== Renamed methods

[cols="1,1,2"]
|===
| core | native | Note

| `isRole(expectedValue?)` → `boolean`
| `hasRoleAttr(expectedValue?)` → `boolean`
| Checks whether the `role` attribute is set

| `isOption(name)` → `boolean`
| `hasOption(name)` → `boolean`
| Checks whether the `<name>-option` attribute is set

| `isReftext()` → `boolean`
| `hasReftext()` → `boolean`
| Checks whether the `reftext` attribute is set
|===

==== Modified return types (asynchronous)

[cols="1,1"]
|===
| core | native

| `getIconUri(name)` → `string`
| `getIconUri(name)` → `Promise<string>` (delegates to `iconUri`)

| `getImageUri(target, key?)` → `string`
| `getImageUri(target, key?)` → `Promise<string>` (delegates to `imageUri`)

| `readAsset(path, opts)` → `string \| undefined`
| `readAsset(path, opts)` → `Promise<string \| null>`

| `readContents(target, opts)` → `void`
| `readContents(target, opts)` → `Promise<string \| null>`

| `normalizeWebPath(...)` → `void`
| `normalizeWebPath(target, start?, preserveUri?)` → `string`

| `normalizeSystemPath(...)` → `void`
| `normalizeSystemPath(target, start?, jail?, opts?)` → `string`

| `normalizeAssetPath(...)` → `void`
| `normalizeAssetPath(assetRef, assetName?, autocorrect?)` → `string`
|===

=== AbstractBlock

==== Asynchronous convert and content

[cols="1,1,2"]
|===
| core | native | Note

| `convert()` → `string`
| `convert()` → `Promise<string>`
| Must be awaited

| `getContent()` → `string`
| `getContent()` → `Promise<string>`
| Alias for `content()` in native; both are asynchronous

| _(no equivalent)_
| `render()` → `Promise<string>`
| Deprecated alias for `convert()`; present for backwards compatibility
|===

==== Titles: asynchronous pre-computation

In `@asciidoctor/core`, `getTitle()` is synchronous and applies substitutions on demand.
In `@asciidoctor/native`, the title is pre-computed asynchronously during parsing via `precomputeTitle()`, then exposed through a synchronous getter.

[cols="1,1"]
|===
| core | native

| `getTitle()` → `string \| undefined` (synchronous, applies subs on every call)
| `get title` → `string \| null` (synchronous, result pre-computed by `precomputeTitle()`)

| _(no equivalent)_
| `async precomputeTitle()` → `Promise<void>`
|===

=== Document

==== Asynchronous parse() and convert()

[cols="1,1"]
|===
| core | native

| `parse(data?)` → `Document` (synchronous)
| `parse(data?)` → `Promise<Document>` (asynchronous)

| `convert(opts?)` → `string` (synchronous)
| `convert(opts?)` → `Promise<string>` (asynchronous)
|===

==== Direct properties instead of getter methods

In `@asciidoctor/native`, several document values are direct instance properties,
accessible without a method call, in addition to the getter methods inherited from the `@asciidoctor/core` API.

[cols="1,1"]
|===
| Getter method (core) | Direct property (native)

| `getSafe()` → `number`
| `document.safe`

| `getCompatMode()` → `boolean`
| `document.compatMode`

| `getSourcemap()` / `setSourcemap(v)` → `boolean`
| `document.sourcemap`

| `getBaseDir()` → `string`
| `document.baseDir`

| `getOutfilesuffix()` → `string`
| `document.outfilesuffix`

| `getBackend()` → `string`
| `document.backend`

| `getDoctype()` → `string`
| `document.doctype`
|===

==== Restructured inner classes

[cols="1,1,2"]
|===
| core | native | Note

| `Document.Title` (inner class)
| `DocumentTitle` (named export)
| Renamed to avoid collision with `Document`

| `Document.Author` (inner class)
| `Author` (named export)
| Top-level export

| `Document.Footnote` (inner class)
| `Footnote` (named export)
| Top-level export

| `Document.ImageReference` (inner class)
| `ImageReference` (named export)
| Top-level export
|===

=== Section

==== Missing setter methods

The following setter methods present in `@asciidoctor/core` are not yet implemented in `@asciidoctor/native`:

[cols="1,2"]
|===
| core method | Note

| `setIndex(index)`
| Sets the 0-based index of the section within its parent

| `setSectionName(name)`
| Sets the section name (e.g., `abstract`, `colophon`)

| `setSpecial(value)`
| Marks the section as a special section
|===

=== Inline

[cols="1,1"]
|===
| core | native

| `convert()` → `string` (synchronous)
| `convert()` → `Promise<string>` (asynchronous)
|===

=== ListItem

==== Asynchronous text pre-computation

In `@asciidoctor/core`, `getText()` applies substitutions synchronously on each call.
In `@asciidoctor/native`, the text is pre-computed during `Document.parse()` via `precomputeText()` and returned synchronously afterwards.

[cols="1,1,2"]
|===
| core | native | Note

| `getText()` → `string` (synchronous, applies subs on each call)
| `getText()` → `string` (synchronous, result pre-computed by `precomputeText()`)
| Return type is the same; underlying mechanism differs

| _(no equivalent)_
| `precomputeText()` → `Promise<void>`
| Called automatically during `Document.parse()`

| `getList()` → `List`
| _(not implemented)_
| Use `getParent()` to retrieve the parent list
|===

=== Table.Cell

==== Asynchronous text pre-computation and content

[cols="1,1,2"]
|===
| core | native | Note

| `getText()` → `string` (synchronous)
| `get text` → `string \| null` (synchronous, result pre-computed by `precomputeText()`)
| Result is computed once during parse; raw text otherwise

| _(no equivalent)_
| `precomputeText()` → `Promise<void>`
| Called automatically during `Document.parse()`

| `getContent()` → `string` (synchronous)
| `content()` → `Promise<string \| string[]>`
| Returns an array for multi-paragraph cells
|===

== Features specific to @asciidoctor/native

These features have no equivalent in `@asciidoctor/core`.

=== Asynchronous pre-computation of titles and reftext

* `AbstractBlock.precomputeTitle()` — asynchronously pre-computes the title with substitutions applied.
* `AbstractNode.precomputeReftext()` — asynchronously pre-computes the reftext with substitutions applied.
* `ListItem.precomputeText()` — asynchronously pre-computes the list item text with substitutions applied.
* `Table.Cell.precomputeText()` — asynchronously pre-computes the cell text with substitutions applied.

These methods are called automatically during `Document.parse()` to allow synchronous getters to work during conversion.

=== Partial browser compatibility

`@asciidoctor/native` uses the `fetch` API for `generateDataUriFromUri` and `readContents` (remote URIs), which works in both Node.js ≥ 18 and modern browsers.
Local file reading methods (`readAsset`, `generateDataUri`) return `null` in environments without `node:fs`.

== Features missing in @asciidoctor/native

These features are present in `@asciidoctor/core` but not yet implemented in `@asciidoctor/native`.

[cols="1,2"]
|===
| Feature | Note

| `Asciidoctor.getRuntime()` → `Object`
| Returns platform / engine / GEM_HOME info; no equivalent concept in a native JS library

| `Section.setIndex(index)` / `setSectionName(name)` / `setSpecial(value)`
| Internal setters used by the parser; not yet exposed as public API

| `ListItem.getList()` → `List`
| Use `getParent()` instead

| `AbstractNode.isUri(str)` (deprecated)
| Present in native but marked `@deprecated` — to be removed in a future version
|===
