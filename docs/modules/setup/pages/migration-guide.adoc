= Migration guide: @asciidoctor/core v3 â†’ v4
:icons: font

`@asciidoctor/core` v4 is a complete rewrite of the library.
v3 relied on Opal (a Ruby-to-JavaScript transpiler) to produce a JS bundle from the Ruby source.
v4 is a native JavaScript implementation â€” no transpilation, no Ruby, pure ESM.

The library source is written in ESM only.
For compatibility, a CommonJS bundle is also provided and the `package.json` exports field points to the appropriate format automatically â€” `require()` continues to work.

== Requirements

* *Node.js >= 22* (LTS)

== Breaking changes

=== Asynchronous API

All parsing and conversion operations now return a `Promise` and must be awaited.

[cols="1,1,2"]
|===
| v3 (Opal) | v4 (native) | Note

| `asciidoctor.load(source, opts)` â†’ `Document`
| `load(source, opts)` â†’ `Promise<Document>`
| Must be awaited

| `asciidoctor.loadFile(path, opts)` â†’ `Document`
| `loadFile(path, opts)` â†’ `Promise<Document>`
| Must be awaited

| `asciidoctor.convert(source, opts)` â†’ `string`
| `convert(source, opts)` â†’ `Promise<string>`
| Must be awaited

| `asciidoctor.convertFile(path, opts)` â†’ `string`
| `convertFile(path, opts)` â†’ `Promise<string>`
| Must be awaited

| `document.parse(data?)` â†’ `Document`
| `document.parse(data?)` â†’ `Promise<Document>`
| Must be awaited

| `document.convert(opts?)` â†’ `string`
| `document.convert(opts?)` â†’ `Promise<string>`
| Must be awaited
|===

=== No factory call

v3 exports a factory function that must be called to obtain an Asciidoctor instance.
v4 uses named exports â€” import only what you need, no factory call needed.

[cols="1,1"]
|===
| v3 | v4

| `require('@asciidoctor/core')()` â†’ must call the factory
| `import { load, convert } from '@asciidoctor/core'` â†’ use functions directly
|===

=== Strict camelCase naming

Some internal properties and options that retained the Ruby `snake_case` style in v3 are now camelCase.

[cols="1,1"]
|===
| v3 (`snake_case`) | v4 (`camelCase`)

| `node_name`
| `nodeName`

| `source_location`
| `sourceLocation`

| `content_model`
| `contentModel`

| `default_subs`
| `defaultSubs`
|===

=== Renamed methods

[cols="1,1,2"]
|===
| v3 | v4 | Note

| `isRole(expectedValue?)` â†’ `boolean`
| `hasRoleAttribute(expectedValue?)` â†’ `boolean`
| Checks whether the `role` attribute is set

| `isOption(name)` â†’ `boolean`
| `hasOption(name)` â†’ `boolean`
| Checks whether the `<name>-option` attribute is set

| `isReftext()` â†’ `boolean`
| `hasReftext()` â†’ `boolean`
| Checks whether the `reftext` attribute is set. Note: v4 also retains `isReftext()` with broader semantics (returns `true` when `reftext` is set *or* the node has a title).

| `getAttr(name, defaultValue?, inherit?)` â†’ `*`
| `getAttribute(name, defaultValue?, inherit?)` â†’ `*`
| `getAttr` is removed. Use `getAttribute` instead â€” the signature is identical.
|===

=== `getAttribute` returns `null` instead of `undefined`

In v3, `getAttribute(name)` returned `undefined` when the attribute was absent.
In v4, it returns `null`.

[cols="1,1"]
|===
| v3 | v4

| `node.getAttribute('missing') === undefined` â†’ `true`
| `node.getAttribute('missing') === null` â†’ `true`
|===

Update any code that checks the return value against `undefined`:

[source,javascript]
----
// v3
if (node.getAttribute('language') === undefined) { ... }

// v4
if (node.getAttribute('language') === null) { ... }
// or equivalently
if (!node.getAttribute('language')) { ... }
----

=== Extensions transpiled with Opal

Extensions written in Ruby and transpiled to JavaScript via Opal will no longer work.
v4 does not include the Opal runtime, so any Opal-generated code will fail at load time.

Extensions must be rewritten in pure JavaScript (ESM).
The extension API is largely similar, but some differences may exist â€” refer to the extension documentation for details.

In addition, extension processors can now be asynchronous: the `process` method may return a `Promise`, which will be properly awaited during conversion.

[cols="1,1"]
|===
| v3 | v4

| Extensions written in Ruby, transpiled via Opal
| Extensions written in pure JavaScript (ESM)
|===

=== Custom converters

In v4, `node.getContent()` returns a `Promise`, so any converter that calls it must `await` the result.
The `convert()` method itself can remain synchronous if it does not call `getContent()` or any other async API.

[source,javascript]
----
// v4 â€” synchronous converter (no async content call)
convert(node, transform) {
  return node.getTitle()
}

// v4 â€” async converter when getContent() is needed
async convert(node, transform) {
  return `<p>${await node.getContent()}</p>`
}
----

=== Custom syntax highlighters

The syntax highlighter API distinguishes between two methods with different async requirements:

* `highlight(node, source, lang, opts)` â€” must remain *synchronous*, returns a `string` or `[string, number]`
* `format(node, lang, opts)` â€” can be synchronous or asynchronous; returning a `Promise<string>` is recommended for I/O-bound operations, but a plain `string` works too

=== CLI: `--require` no longer registers extensions

This change applies to the `asciidoctor` CLI package.

In v3, `--require` (`-r`) loaded the given module and automatically called its exported `register` function, so it was the way to load extensions from the command line.
In v4, these two concerns are separate options:

* `--require <library>` (`-r`) â€” loads the module for its side effects only, without calling any exported function.
Use it for libraries that configure themselves on load (syntax highlighters, polyfills, etc.) or for extensions that self-register at the top level via `Extensions.register()`.
* `--extension <extension>` â€” loads the module and calls its `register(registry)` named export with a shared registry.
Repeat the option to load multiple extensions.

For example, to load the https://github.com/asciidoctor/asciidoctor-kroki[asciidoctor-kroki] extension:

[source,console]
----
$ asciidoctor -r asciidoctor-kroki doc.adoc <1>
$ asciidoctor --extension asciidoctor-kroki doc.adoc <2>
----
<1> v3
<2> v4
